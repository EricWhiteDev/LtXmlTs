Here is a bug:
- Default namespace not consulted when auto-generating p# prefixes

Lines 944 and 954 — populateNamespacePrefixInfoRecurse has two places that decide whether to auto-generate a p# prefix declaration. Both check only namespacePrefixPairs; neither checks info.defaultNamespace:

  // Line 944 — for attribute namespaces:
  const alreadyMapped = info.namespacePrefixPairs.some(p => p.namespace === attr.name.namespace);

  // Line 954 — for element name namespace:
  const alreadyMapped = info.namespacePrefixPairs.some(p => p.namespace === element.name.namespace);

  When an element's namespace matches the default namespace declared by xmlns='...' on an ancestor, info.defaultNamespace holds that namespace but it is never checked. The code sees no
  matching pair, sets alreadyMapped = false, and generates a redundant xmlns:p0='...' declaration and prefix.

  Concrete example:
  const ns = new XNamespace('urn:example');
  const root = new XElement(ns + 'root',
    new XAttribute(XNamespace.xmlns + 'xmlns', 'urn:example')
  );
  root.toString();
  // Current:  <p0:root xmlns='urn:example' xmlns:p0='urn:example' />
  // Correct:  <root xmlns='urn:example' />

  The fix is to add || element.name.namespace === info.defaultNamespace to the alreadyMapped check on line 954 (element names). Note that line 944 (attribute namespaces) does not need a
  similar fix — XML attributes cannot be in the default namespace; unprefixed attributes are always in no namespace, and the XNamespace.none guard on line 943 already skips them.

add unit tests.
