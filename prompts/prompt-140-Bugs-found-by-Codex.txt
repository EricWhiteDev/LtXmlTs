Findings:

  1. High: parsing drops significant whitespace text nodes inside elements. In src/SaxParser.ts:185, ontext returns early for any /^\s*$/ text. That means XElement.parse('<root> <a/>

     loss, not just formatting loss.
  2. High: inserting an XElement that already belongs to another parent breaks tree ownership invariants. In src/XContainer.ts:116, element content is always appended directly and its

  3. Medium: removing a foreign attribute corrupts the attribute list instead of doing nothing or throwing. In src/XElement.ts:190, indexOf is used without checking -1. Calling

  4. Medium: removing a foreign child node corrupts the child list the same way. In src/XContainer.ts:367, removeChild() also assumes the node exists. Calling it with a node that is

 Context

 Full review of the LtXmlTs (LINQ to XML for TypeScript) library source code to identify
 bugs. All 1105 tests currently pass, so these bugs are in code paths not fully covered by
  tests.

 Bug 1: XElement nodes not cloned when they already have a parent (CRITICAL)

 File: src/XContainer.ts:116-121

 if ((content as { nodeType?: string })?.nodeType === "Element") {
   const el = content as XNode;
   el.parent = this;
   this.nodesArray.push(el);
   return;
 }

 Problem: Unlike XComment, XText, XCData, and XProcessingInstruction (handled at lines
 91-115), XElement nodes are NOT cloned when they already have a parent. This silently
 corrupts the tree: the old parent still references the node in its nodesArray, but the
 node's parent now points to the new container. In .NET LINQ to XML, adding a node that
 already has a parent creates a deep clone.

 Fix: Add a parent check and clone XElement like the other node types:
 if ((content as { nodeType?: string })?.nodeType === "Element") {
   const el = content as XNode;
   if (el.parent === null) {
     el.parent = this;
     this.nodesArray.push(el);
   } else {
     const clone = new XElement(el as unknown as XElement);
     clone.parent = this;
     this.nodesArray.push(clone);
   }
   return;
 }

 Bug 2: removeChild corrupts array when child not found

 File: src/XContainer.ts:367-370

 public removeChild(child: XNode): void {
   const idx = this.nodesArray.indexOf(child);
   this.nodesArray = [...this.nodesArray.slice(0, idx), ...this.nodesArray.slice(idx +
 1)];
 }

 Problem: When indexOf returns -1, slice(0, -1) removes the last element and slice(0)

 public removeChild(child: XNode): void {
   const idx = this.nodesArray.indexOf(child);
   if (idx === -1) {
     return;
   }
   this.nodesArray = [...this.nodesArray.slice(0, idx), ...this.nodesArray.slice(idx +
 1)];
 }
