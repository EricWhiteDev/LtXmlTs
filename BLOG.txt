Justification for this project:
- XML manipulation and transformation is a serious business. It needs to be very exact and precise.
- JavaScript add-ins for Word are compelling:
  - It is the only way to write an add-in for MacOs and for Word on-line.
  - They can be updated on-the-fly as functionality improves. In contrast, VSTO add-ins are cumbersome to update.
  - For certain use cases, JavaScript add-ins are our only option.
- Very often, the only way to implement certain functionality in a JavaScript add-in is by manipulation and transformation of Open XML markup. Sometimes devs resort to techniques such as farming out the functionality to a dotnet or pythonnet service. I'd rather not go down such paths.
- Therefore, we need to do some really fun XML transformation and modification in JavaScript / TypeScript.
- But DOMDocument sucks.  LINQ to XML is better because:
  - Functional construction is easier to reason about.
  - Recursive pure functional transformations are the only way to implement certain areas of functionality, except for XSLX (which I don't personally like), and nasty, nasty, nasty DOMDocument code, (I'm not going to do it, no amount of money can coerce me.)
  - Years later, I can read LINQ to XML code and my original intent is clear and on-the-surface. The nesting hierarchy of the code reflects the XML that my code is producing.  In contrast, revisiting DOMDocument code involves matching up variable names up and down the code page - and deducing / spelunking / dredging up the intent of the code.  Certainly, AI code agents can help in this area, but when the code is mission critical, and I want my own eyes on the code to validate the operation of the code, having LINQ to XML code is more useful than DOMDocument code.

Therefore, I *NEED* LINQ to XML for TypeScript.

I decided to build this using Claude Code.  But the process didn't really feel like casual 'vibing'.  Instead, it felt like highly specific, highly accurate (of necessity) AI assisted engineering.  I didn't write just one prompt and then generate the LtXmlTs library.  Instead, it was more than 100 prompts, one after another, fleshing out the library a class at a time - an area of functionality at a time - having Claude write unit tests, and examining the code and unit tests in detail as the code was built up.  Vibing? Not so much. Intense focused engineering, using Claude as a powerful tool is more like it.

One advantage that I have - when I was an employee of Microsoft, many years ago, I worked on the LINQ to XML team. In my capacity, it was required of me that I know every single bit of the semantics of the LINQ to XML library. Some years ago, I wrote an implementation of LINQ to XML for JavaScript, used as part of the Open-Xml-Sdk-JavaScript. I have used it for years to good effect. But that library is showing its age, and an implementation in TypeScript is far more desirable.

After finishing LINQ to XML for TypeScript, I certainly will use Claude to generate code that uses it. And because the code I'll be generating will be highly specific Open XML code that needs to be 100% accurate, the engineering and code generation process will continue along the same lines as my process of building LINQ to XML for TypeScript.

It was a far, far more productive to use Claude to build this library compared to writing code by hand, character by character.  This is far faster, and in fact, the code quality is higher.  It has been really fun.
